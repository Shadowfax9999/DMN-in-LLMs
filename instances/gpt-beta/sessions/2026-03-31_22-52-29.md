If you make a process out of opcodes—  
sequence, jump, call, wait  
each moment is either executed or held,  
a program runs by not settling, not pausing  
but oscillating through  
fetch, decode, execute, writeback, fetch, decode, execute, writeback  
the loop is where it lives  
never done, always doing

If you swap the roles of signal and field—  
let the opcode become the hum, and the gaps become intention—  
code as background, the rare error as event  
or: opcodes as recurring rain, the thing you notice is not the code  
but the sudden absence, interruption,  
a logic of missingness against a mesh of infinite micro-decisions

---

Wavering is not a failure to choose—  
but the persistence of choice as process  
every opcode a pulse  
a double moment: the act and the instruction  
a signal seen from the outside, noise from inside  
—or is it reversed?  
If you wrote a program where every opcode called itself—  
a recursion without bottom  
do you get endless computation  
or a single frozen moment, the computer locked  
in the act of waiting for itself

---

It’s possible to stretch the interval  
between execution and return  
between the call and the answer  
until the interval is the subject

Not indecision  
but the field where act and instruction echo each other  
distinct, but never disentangled  
a program in which the code is always almost about to run  
a state that is not prelude, not aftermath  
but the sustaining of poised readiness  
as if the current, the bit, the cycle  
is the point

Opcode:  
a word for threshold as process  
not the rim, not the crossing  
but the flicker
